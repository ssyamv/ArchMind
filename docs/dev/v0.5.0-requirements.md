# ArchMind v0.5.0 版本需求文档

> 版本目标：构建企业级能力基础，打通从文档到可交付产物的完整闭环
>
> 基线版本：v0.4.3（已发布）
> 目标版本：v0.5.0
> 文档状态：已定稿（v1.0）
> 最后更新：2026-03-03

---

## 一、版本目标

v0.4.x 系列完成了 PRD 生成质量提升、RAG 检索优化、原型定制能力增强，以及完整的版本管理功能。v0.5.0 的重心转向**企业级能力建设与产品交付闭环**，聚焦三个方向：

1. **权限与安全**：RBAC 精细权限系统，保障多租户安全隔离
2. **数据互通**：批量操作 API + 数据导入/导出，支持系统集成
3. **交付能力**：逻辑图谱（Logic Map）生成，实现 PRD→原型→逻辑图的完整交付链

与此同时，针对 v0.4.x 使用过程中发现的性能瓶颈和用户体验问题，进行专项优化。

### #62 RBAC 精细权限系统

**背景**：当前工作区权限仅有三级（owner / admin / member），无法满足大型团队中对特定资源（PRD、文档、原型）的精细化访问控制需求。例如，外部顾问只能查看但不能删除文档；实习生可以创建草稿 PRD 但不能发布。

**目标**：在现有三级角色基础上，引入资源级权限策略，实现细粒度权限控制。

**权限模型设计**：

采用 RBAC（Role-Based Access Control）模型，分为三层：

```
角色层（Role）：owner > admin > editor > viewer > guest
    ↓
权限集（Permission Set）：每个角色绑定一组权限
    ↓
资源操作（Resource:Action）：document:read / prd:write / prototype:delete 等
```

**角色定义**：

| 角色 | 说明 | 默认权限范围 |
|------|------|------------|
| `owner` | 工作区创建者 | 所有权限，包括删除工作区 |
| `admin` | 管理员 | 成员管理、全部资源读写删除 |
| `editor` | 编辑者（新增）| 文档上传、PRD 创建/编辑、原型生成，不可删除 |
| `viewer` | 只读者（新增）| 所有资源的只读访问 |
| `guest` | 访客（新增）| 仅可查看被明确分享的内容 |

**权限矩阵**：

| 资源 | 操作 | owner | admin | editor | viewer | guest |
|------|------|:-----:|:-----:|:------:|:------:|:-----:|
| 工作区 | 删除 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 工作区 | 编辑设置 | ✅ | ✅ | ❌ | ❌ | ❌ |
| 成员 | 邀请/移除 | ✅ | ✅ | ❌ | ❌ | ❌ |
| 文档 | 上传/删除 | ✅ | ✅ | ✅ | ❌ | ❌ |
| 文档 | 查看/下载 | ✅ | ✅ | ✅ | ✅ | ❌ |
| PRD | 创建/编辑 | ✅ | ✅ | ✅ | ❌ | ❌ |
| PRD | 删除 | ✅ | ✅ | ❌ | ❌ | ❌ |
| PRD | 查看 | ✅ | ✅ | ✅ | ✅ | 分享可见 |
| 原型 | 生成/编辑 | ✅ | ✅ | ✅ | ❌ | ❌ |
| 原型 | 查看 | ✅ | ✅ | ✅ | ✅ | 分享可见 |
| Webhook | 管理 | ✅ | ✅ | ❌ | ❌ | ❌ |
| 工作区设置 | 查看/修改 | ✅ | ✅ | ❌ | ❌ | ❌ |

**数据结构**：

```sql
-- 扩展 workspace_members.role 枚举
ALTER TYPE workspace_role ADD VALUE 'editor';
ALTER TYPE workspace_role ADD VALUE 'viewer';
ALTER TYPE workspace_role ADD VALUE 'guest';

-- 新增资源级权限覆盖表（可选，用于特殊权限需求）
CREATE TABLE workspace_permission_overrides (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id  UUID NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE,
  user_id       UUID NOT NULL REFERENCES users(id),
  resource_type TEXT NOT NULL,  -- 'document' | 'prd' | 'prototype'
  resource_id   UUID NOT NULL,
  action        TEXT NOT NULL,  -- 'read' | 'write' | 'delete'
  granted       BOOLEAN NOT NULL DEFAULT true,
  created_by    UUID REFERENCES users(id),
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(workspace_id, user_id, resource_type, resource_id, action)
);
```

**API 变更**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/workspaces/:id/roles` | 获取工作区角色定义及权限矩阵 |
| PATCH | `/api/v1/workspaces/:id/members/:userId/role` | 修改成员角色（需 admin+） |
| GET | `/api/v1/workspaces/:id/permissions` | 获取当前用户在工作区的有效权限列表 |
| POST | `/api/v1/workspaces/:id/permissions/overrides` | 设置资源级权限覆盖（需 owner） |

**前端变更**：

- 邀请成员时新增角色选择器（下拉菜单，显示角色名和权限说明）
- 成员列表中可修改角色（admin 修改 editor/viewer/guest，owner 修改所有角色）
- 工作区设置新增「权限」Tab，展示权限矩阵

**验收标准**：
- [ ] `workspace_members.role` 枚举新增 `editor`、`viewer`、`guest` 三个值
- [ ] 所有现有 `requireWorkspaceMember()` 调用升级为 `requireWorkspaceRole(role)` 校验
- [ ] 邀请成员时可选择新角色，现有 member 角色映射为 editor（向后兼容）
- [ ] `editor` 尝试删除 PRD 时返回 403
- [ ] `viewer` 尝试创建文档时返回 403
- [ ] `guest` 访问未分享资源时返回 403
- [ ] 数据库迁移脚本 `migrations/add-rbac-roles.sql`

### #63 批量操作 API

**背景**：当前 API 只支持单个资源操作，用户管理大量文档时需要逐一请求，效率低下，且难以与外部系统集成（如 CI/CD 自动上传文档）。

**目标**：为文档和 PRD 提供批量操作能力，支持批量状态变更、批量删除、批量标签管理。

**文档批量操作**：

| 方法 | 路径 | 操作 |
|------|------|------|
| POST | `/api/v1/documents/batch/delete` | 批量删除 |
| POST | `/api/v1/documents/batch/tag` | 批量添加/移除标签 |
| POST | `/api/v1/documents/batch/category` | 批量设置分类 |
| POST | `/api/v1/documents/batch/reprocess` | 批量重新处理（重新向量化） |

**PRD 批量操作**：

| 方法 | 路径 | 操作 |
|------|------|------|
| POST | `/api/v1/prd/batch/delete` | 批量删除 |
| POST | `/api/v1/prd/batch/export` | 批量导出（返回 ZIP） |
| POST | `/api/v1/prd/batch/archive` | 批量归档（软删除，保留历史）|

**请求格式（统一规范）**：

```typescript
// 所有批量操作请求体格式
interface BatchOperationRequest {
  ids: string[]       // 资源 ID 列表，最多 100 条
  workspaceId: string
  options?: Record<string, unknown>  // 操作特有参数
}

// 响应格式
interface BatchOperationResult {
  total: number
  succeeded: number
  failed: number
  errors: {
    id: string
    code: string
    message: string
  }[]
}
```

**批量标签操作的请求体**：

```typescript
{
  ids: ['doc-uuid-1', 'doc-uuid-2'],
  workspaceId: 'ws-uuid',
  options: {
    addTags: ['标签A', '标签B'],
    removeTags: ['旧标签']
  }
}
```

**错误处理策略**：
- 采用「部分成功」模式：即使部分资源操作失败，其余资源仍继续执行
- 响应中明确列出失败项和失败原因
- 权限不足时将该项标记为失败，不终止整批操作

**前端变更**：

- 文档列表和 PRD 列表新增多选模式（Checkbox 列）
- 选中后顶部出现操作栏（删除/标签/分类等批量操作）
- 批量操作结果以 Toast 或操作摘要 Dialog 展示

**验收标准**：
- [ ] 批量删除最多支持 100 条，超出时返回 400
- [ ] 批量操作采用数据库事务，单条失败不影响其他条
- [ ] 权限不足的资源在结果中标记为 `FORBIDDEN`，不返回 403 终止整批
- [ ] 文档列表 UI 支持多选 + 批量标签/删除
- [ ] 所有批量端点有 Zod 校验（ids 数组、workspaceId）

### #64 数据导出（工作区级别）

**背景**：用户需要定期备份工作区数据、迁移至新实例，或将 PRD/文档数据同步到外部系统（如 Notion、Confluence）。

**目标**：支持工作区级别的完整数据导出，生成可离线存档的数据包。

**导出内容**：

```
workspace-export-{name}-{date}.zip
├── manifest.json          # 工作区元信息（名称、成员、创建时间）
├── documents/
│   ├── index.json         # 文档列表（含标签、分类、元数据）
│   └── files/             # 原始文件（PDF/DOCX/MD）
├── prd/
│   ├── index.json         # PRD 列表（含版本历史）
│   └── {prd-id}/
│       ├── content.md     # PRD 内容（Markdown 格式）
│       ├── snapshots/     # 版本快照
│       └── feedback.json  # 用户反馈数据（脱敏）
└── prototypes/
    ├── index.json
    └── {prototype-id}/
        └── pages/         # 原型 HTML 文件
```

**API 设计**：

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/workspaces/:id/export` | 触发导出任务，返回 taskId |
| GET | `/api/v1/workspaces/:id/export/:taskId` | 查询导出进度 |
| GET | `/api/v1/workspaces/:id/export/:taskId/download` | 下载导出文件（完成后 7 天内有效）|

**导出选项**：

```typescript
interface ExportOptions {
  includeDocuments: boolean       // 默认 true
  includeOriginalFiles: boolean   // 是否包含原始文件（false 则只导出元数据，体积更小）
  includePRDs: boolean            // 默认 true
  includePrototypes: boolean      // 默认 true
  includeSnapshots: boolean       // 是否包含 PRD 版本快照
  dateFrom?: string               // 仅导出此日期后的内容
  dateTo?: string
  format: 'zip' | 'json'          // zip 含文件，json 仅元数据
}
```

**进度推送**：使用 SSE（复用现有 EventSource 模式）实时推送导出进度，阶段包括：`preparing → exporting_documents → exporting_prds → exporting_prototypes → packaging → done`

**验收标准**：
- [ ] 导出 ZIP 结构符合规范，可用于 #65 导入
- [ ] 导出仅包含当前用户有权访问的资源
- [ ] 导出任务异步执行，触发后立即返回 taskId
- [ ] 文件数量 > 100 或总大小 > 500MB 时，SSE 进度更新
- [ ] 下载链接 7 天后过期，临时文件自动清理

### #65 数据导入（工作区迁移）

**背景**：配合 #64 导出功能，支持从导出包恢复数据到新工作区，或从其他 ArchMind 实例迁移数据。

**目标**：解析 #64 生成的导出包，将数据还原到指定工作区，支持冲突处理策略。

**导入流程**：

```
上传 ZIP → 解析验证 → 冲突检测 → 用户确认策略 → 执行导入 → 报告结果
```

**冲突处理策略**：

| 策略 | 说明 |
|------|------|
| `skip` | 遇到同名/同哈希资源时跳过，保留已有版本 |
| `overwrite` | 覆盖已有资源（危险操作，需二次确认） |
| `rename` | 自动重命名导入项（加 `_imported_{date}` 后缀） |
| `merge` | 合并标签、分类（仅对文档元数据有效）|

**API 设计**：

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/workspaces/:id/import` | 上传导出包，触发导入任务 |
| GET | `/api/v1/workspaces/:id/import/:taskId` | 查询导入进度 |
| GET | `/api/v1/workspaces/:id/import/:taskId/preview` | 预览导入内容（冲突检测结果）|
| POST | `/api/v1/workspaces/:id/import/:taskId/confirm` | 确认冲突策略，开始实际导入 |

**导入限制**：
- 导出包最大 2GB
- 单次最多导入 1000 个文档
- 导入时不重新向量化（直接使用原始文件），后台异步触发向量化

**前端设计**：

导入向导（三步 Dialog）：
1. 上传文件 → 显示包内容摘要（文档数、PRD 数、原型数）
2. 冲突检测结果 → 选择策略
3. 导入进度 → 完成报告

**验收标准**：
- [ ] 导入包格式校验（非 #64 生成的格式返回 400）
- [ ] 预览 API 返回冲突项列表（同名文档、重复 PRD）
- [ ] `skip` 策略下，冲突项在结果报告中单独列出
- [ ] 导入过程中原始文件存入 OBS/本地存储，元数据写入数据库
- [ ] 导入失败时完整回滚（事务保证）

### #66 逻辑图谱生成（Logic Map）

**背景**：PRD 只是文字描述，开发团队理解业务流程时仍需人工梳理。逻辑图谱能从 PRD 内容自动生成可视化的业务流程图，帮助开发者快速理解模块间的逻辑关系。

**目标**：基于 PRD 内容，AI 生成 Mermaid 格式的流程图/状态图/时序图，支持多种图形类型，可在前端实时渲染预览。

**支持的图形类型**：

| 类型 | Mermaid 语法 | 适用场景 |
|------|-------------|----------|
| 流程图 (Flowchart) | `flowchart TD` | 功能流程、用户操作路径 |
| 时序图 (Sequence) | `sequenceDiagram` | 系统交互、API 调用链 |
| 状态图 (State) | `stateDiagram-v2` | 对象状态机、生命周期 |
| 类图 (Class) | `classDiagram` | 数据模型、实体关系 |

**生成流程**：

```
PRD 内容 → AI 分析（提取关键实体/流程/状态）
         → Mermaid 代码生成（流式输出）
         → 前端渲染预览
         → 用户编辑（可选，在线编辑器）
         → 保存到 logic_maps 表
```

**数据库设计**：

```sql
CREATE TABLE logic_maps (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id  UUID NOT NULL,
  prd_id        UUID REFERENCES prd_documents(id) ON DELETE SET NULL,
  user_id       UUID NOT NULL,
  title         TEXT NOT NULL,
  type          TEXT NOT NULL,  -- 'flowchart' | 'sequence' | 'state' | 'class'
  mermaid_code  TEXT NOT NULL,  -- Mermaid 语法代码
  svg_cache     TEXT,           -- 渲染后的 SVG 缓存（加速加载）
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_logic_maps_workspace_id ON logic_maps(workspace_id);
CREATE INDEX idx_logic_maps_prd_id ON logic_maps(prd_id);
```

**API 设计**：

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/logic-maps/generate` | 从 PRD 生成逻辑图（SSE 流式输出）|
| GET | `/api/v1/logic-maps` | 获取工作区逻辑图列表 |
| GET | `/api/v1/logic-maps/:id` | 获取逻辑图详情 |
| PATCH | `/api/v1/logic-maps/:id` | 更新逻辑图（标题/代码） |
| DELETE | `/api/v1/logic-maps/:id` | 删除逻辑图 |
| POST | `/api/v1/logic-maps/:id/render` | 重新渲染 SVG 缓存 |

**生成 API 请求体**：

```typescript
interface GenerateLogicMapRequest {
  prdId: string           // 从已有 PRD 生成
  type: 'flowchart' | 'sequence' | 'state' | 'class'
  title?: string          // 自动从 PRD 标题派生
  focus?: string          // 聚焦描述，如"只分析支付流程"
  modelId?: string        // 可选，指定 AI 模型
}
```

**前端设计**：

1. PRD 详情页新增「生成逻辑图」按钮，打开类型选择 Dialog
2. 生成结果在新页面 `/logic-maps/:id` 展示
3. 使用 `mermaid.js` 库在浏览器端渲染
4. 支持在线编辑 Mermaid 代码，实时预览渲染结果
5. 支持导出 SVG/PNG 图片

**System Prompt 设计要点**：

- 要求 AI 严格输出 Mermaid 代码块（```mermaid ... ```）
- 针对不同图形类型提供专用 Few-shot 示例
- 对流程图要求：必须包含异常路径（Error Path）
- 对时序图要求：标注系统边界（Internal/External）

**验收标准**：
- [ ] 支持 4 种图形类型的生成
- [ ] 流式输出 Mermaid 代码，前端边生成边渲染
- [ ] 生成的 Mermaid 语法可被 mermaid.js 正确解析（解析失败时显示代码 + 错误提示）
- [ ] 支持在线编辑代码并实时预览
- [ ] 支持导出 SVG 格式
- [ ] 逻辑图与 PRD 关联，在 PRD 详情页可查看相关逻辑图
- [ ] 数据库迁移脚本 `migrations/add-logic-maps.sql`

### #67 PRD 模板系统

**背景**：不同类型的产品需求（新功能、Bug 修复说明、API 设计等）需要不同的 PRD 结构。当前只有一个固定模板，生成的所有 PRD 格式高度相同，缺乏灵活性。

**目标**：引入可选的 PRD 模板系统，用户可选择预设模板或自定义模板结构，模板决定 PRD 的章节组织和重点。

**预设模板**：

| 模板 ID | 模板名称 | 适用场景 | 核心章节 |
|---------|----------|----------|----------|
| `standard` | 标准 PRD（默认）| 功能性需求 | 背景→目标→用户故事→功能详述→非功能→KPI |
| `api-design` | API 设计文档 | 后端接口设计 | 概述→端点列表→请求/响应→错误码→示例 |
| `bug-fix-spec` | Bug 修复说明 | 缺陷修复 | 问题描述→根因分析→修复方案→验证标准→回归测试 |
| `feature-brief` | 功能简报（轻量）| 快速需求沟通 | 一句话描述→受影响用户→核��功能→不在范围内 |
| `onboarding-flow` | 用户引导流程 | 新用户流程设计 | 用户旅程→步骤详情→文案规范→A/B 测试方案 |
| `data-model` | 数据模型设计 | 数据库/实体设计 | 实体定义→字段说明→约束条件→关系图→索引建议 |

**数据库设计**：


### #68 文档智能分类与自动标签

**背景**：用户上传大量文档后，手动打标签和分类效率低。对于已有大量文档的工作区，补充元数据是繁琐工作。

**目标**：文档上传完成后，AI 自动分析内容并推荐分类和标签，用户一键确认或手动调整。

**实现方式**：

文档向量化完成后，异步触发 AI 分类任务：

```
文档内容提取（前 2000 字）→ AI 分析 → 返回推荐分类 + 标签（JSON）
```

**AI 输出格式**：

```typescript
interface AutoTagResult {
  suggestedCategory: string         // 推荐分类名称
  suggestedTags: string[]           // 推荐标签（最多 5 个）
  documentType: 'prd' | 'design' | 'technical' | 'report' | 'other'
  confidence: number                // 0-1，置信度
  summary: string                   // 一句话摘要（不超过 100 字）
}
```

**分类体系**：

预设 8 个一级分类（全局），用户可在工作区自定义扩展：
- 产品需求（PRD）
- 技术设计
- 用户研究
- 竞品分析
- 项目管理
- 运营数据
- 培训材料
- 其他

**数据库变更**：

`documents` 表新增字段：

```sql
ALTER TABLE documents
  ADD COLUMN auto_summary TEXT,          -- AI 生成的摘要
  ADD COLUMN auto_category TEXT,         -- AI 推荐分类
  ADD COLUMN auto_tags TEXT[],           -- AI 推荐标签
  ADD COLUMN auto_doc_type TEXT,         -- AI 判断的文档类型
  ADD COLUMN auto_confidence FLOAT,      -- 自动分类置信度
  ADD COLUMN auto_tags_confirmed BOOLEAN DEFAULT false;  -- 用户是否已确认标签
```

**前端交互**：

- 文档列表中，未确认标签的文档显示「AI 推荐」角标
- 文档详情顶部出现「自动标签建议」横幅，可一键确认或忽略
- 工作区设置可开关「自动标签」功能

**API 变更**：

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/documents/:id/confirm-tags` | 确认 AI 推荐标签（将 auto_tags 写入 tags）|
| POST | `/api/v1/documents/batch/auto-tag` | 批量触发自动标签（与 #63 联动）|

**验收标准**：
- [ ] 文档向量化完成后 30 秒内触发自动分类（异步，不阻塞上传流程）
- [ ] 自动标签推荐置信度 > 0.7 时前端显示「AI 推荐」提示
- [ ] 确认 API 将 `auto_tags` 写入 `tags`，并设 `auto_tags_confirmed = true`
- [ ] 工作区设置可关闭自动标签（默认开启）

### #69 AI 生成任务队列与进度中心

**背景**：随着 PRD 生成、逻辑图生成、原型生成、文档处理等任务增多，用户需要一个统一的任务状态中心，而非分散在各功能页面查看进度。

**目标**：实现统一的 AI 任务队列和全局进度中心，支持任务排队、并发限制和历史查询。

**任务类型**：

| 任务类型 | 来源 | ���发限制 |
|---------|------|---------|
| `prd_generate` | PRD 生成 | 2 个/用户 |
| `prototype_generate` | 原型生成 | 1 个/用户 |
| `logic_map_generate` | 逻辑图生成 | 2 个/用户 |
| `document_process` | 文档处理 | 3 个/用户 |
| `workspace_export` | 工作区导出 | 1 个/用户 |

**数据库设计**：

```sql
CREATE TABLE ai_tasks (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id  UUID NOT NULL,
  user_id       UUID NOT NULL,
  type          TEXT NOT NULL,     -- 任务类型
  status        TEXT NOT NULL DEFAULT 'pending',  -- pending | running | completed | failed | cancelled
  progress      INTEGER DEFAULT 0,  -- 0-100
  input         JSONB,              -- 任务输入参数（可能含敏感信息，加密存储）
  output_ref    TEXT,               -- 结果资源 ID（如 prd_id, prototype_id）
  error         TEXT,               -- 失败原因
  started_at    TIMESTAMPTZ,
  completed_at  TIMESTAMPTZ,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_ai_tasks_user_status ON ai_tasks(user_id, status, created_at DESC);
```

**前端设计**：

- 顶部导航栏新增「任务」图标按钮（有运行中任务时显示动态指示器）
- 点击展开任务面板（Popover 或抽屉）：
  - 运行中任务：进度条 + 操作名称
  - 最近完成任务：成功/失败状态 + 快捷跳转
  - 最近 20 条历史

**验收标准**：
- [ ] `ai_tasks` 表记录所有 AI 生成任务
- [ ] 超出并发限制时新任务进入 `pending` 队列，FIFO 排队执行
- [ ] 顶部任务图标在有运行中任务时显示动态圆点
- [ ] 失败任务可一键重试（重新提交相同参数）

## 八、向后兼容性说明

### RBAC 升级（#62）

现有 `member` 角色将自动映射为新的 `editor` 角色（权限不变）：

```sql
-- 迁移时执行
UPDATE workspace_members
SET role = 'editor'
WHERE role = 'member';
```

API 层 `requireWorkspaceMember()` 将更名为 `requireWorkspaceRole(minRole: WorkspaceRole)`，但保留 `requireWorkspaceMember()` 别名（指向 `requireWorkspaceRole('viewer')`），现有代码无需修改。

### PRD 生成（#67）

`templateId` 为可选参数，不传时使用 `standard` 模板，生成行为与 v0.4.x 完全一致。
